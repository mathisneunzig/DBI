\section{SQL Optimization}

% ------------------------------------------------------------
\begin{frame}{Why Query Optimization Matters}
  A correct query that runs in 30 seconds is useless in a web application that
  must respond in under 200\,ms. As data volumes grow, the difference between
  an optimized and an unoptimized query can be the difference between seconds
  and hours.

  \medskip
  \begin{block}{Two Levels of Optimization}
    \begin{enumerate}
      \item \textbf{Query-level}: rewrite the SQL to express the same result
            more efficiently -- reduce the number of rows processed as early
            as possible.
      \item \textbf{Schema-level}: add or adjust \textbf{indexes} so the
            database engine can locate the required rows without scanning the
            entire table.
    \end{enumerate}
  \end{block}

  \begin{alertblock}{The golden rule}
    \textbf{Measure before you optimize.} Use \texttt{EXPLAIN} / \texttt{EXPLAIN
    ANALYZE} to find the real bottleneck -- do not guess. Optimizing the wrong
    query wastes time and can even make things worse.
  \end{alertblock}
\end{frame}

% ------------------------------------------------------------
\begin{frame}{How a Query is Executed -- The Query Planner}
  When you send a \texttt{SELECT} statement, the DBMS does not execute it
  literally. Instead, a \textbf{query planner (optimizer)} produces an
  \emph{execution plan} -- a tree of physical operations that produce the
  same result as efficiently as possible.

  \medskip
  \begin{block}{Typical plan steps}
    \begin{enumerate}
      \item \textbf{Parsing}: validate syntax, resolve table/column names
      \item \textbf{Logical rewriting}: push predicates down, eliminate
            redundancy, reorder joins
      \item \textbf{Statistics lookup}: the planner uses \emph{table
            statistics} (row count, column cardinality, histogram) to
            estimate costs
      \item \textbf{Physical plan selection}: choose between full table
            scan, index scan, nested-loop join, hash join, etc.
      \item \textbf{Execution}: run the chosen plan and stream results
    \end{enumerate}
  \end{block}

  \begin{examples}{Key insight}
    The planner can only choose from the access paths that \emph{exist}.
    Adding an index creates a new, cheaper access path for the planner to
    choose.
  \end{examples}
\end{frame}

% ------------------------------------------------------------
\begin{frame}[fragile]{EXPLAIN and EXPLAIN ANALYZE}
  \begin{block}{Inspecting the Execution Plan}
\begin{lstlisting}[style=sqlstyle]
-- Show the query plan (estimated costs, no actual execution)
EXPLAIN
SELECT * FROM product WHERE brand_id = 5;

-- Execute the query AND show actual timing + row counts
EXPLAIN ANALYZE
SELECT p.name, b.name
FROM   product p
JOIN   brand b ON p.brand_id = b.id
WHERE  p.price > 10;
\end{lstlisting}
  \end{block}

  \begin{block}{What to look for in the output}
    \begin{itemize}
      \item \textbf{Full Table Scan} (\texttt{ALL} in MySQL): the engine
            reads every row -- a red flag for large tables.
      \item \textbf{Index Scan / Index Range Scan}: the engine navigates
            an index tree -- much faster.
      \item \textbf{rows}: estimated number of rows examined.
            A high value with no index means a slow query.
      \item \textbf{Extra: Using filesort}: sorting without an index --
            consider adding an index on the \texttt{ORDER BY} column.
    \end{itemize}
  \end{block}
\end{frame}
